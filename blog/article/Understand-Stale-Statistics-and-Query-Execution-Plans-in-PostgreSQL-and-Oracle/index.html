1.

What Are Database Statistics

Database statistics are metadata that the query optimizer uses to estimate the cost of different execution plans. These statistics include.

  • Row counts in tables
  • Distribution of values in columns
  • Index cardinality
  • Data density and selectivity
  • NULL value frequencies

When these statistics become outdated (stale), the optimizer makes suboptimal decisions, leading to poor query performance.

2.

Stale Statistics in PostgreSQL

2.1.

How Statistics Affect Execution Plans

PostgreSQL's query planner uses statistics from the pg_statistic table to create efficient execution plans. When statistics are stale, the planner may.

  • Choose nested loop join when hash join would be better
  • Estimate far fewer rows than actually returned
  • Use wrong index or no index at all
  • Allocate insufficient memory for operations

2.2.

Detect Stale Statistics

2.2.1.

Check the pg_stat_all_tables view

1SELECT 
2  schemaname,
3  relname,
4  n_live_tup,
5  n_dead_tup,
6  n_mod_since_analyze,
7  last_autoanalyze,
8  last_analyze
9FROM pg_stat_all_tables
10WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
11  AND n_mod_since_analyze > 0.1 * n_live_tup  -- More than 10% rows modified
12ORDER BY n_mod_since_analyze DESC;

Key indicators.

  • n_mod_since_analyze - rows modified since last analysis

  • n_live_tup - current live row count

  • Large n_dead_tup combined with no recent VACUUM suggests stale stats

When to worry.

  • n_mod_since_analyze > 10% of n_live_tup (indicates significant data changes)

  • last_analyze is days or weeks old on frequently modified tables

  • Query performance suddenly degrades without code changes

2.2.2.

Compare execution plan with actual stats

1-- Get execution plan estimate
2EXPLAIN SELECT * FROM "Session" WHERE "userId" = 'abc123';
3-- Output shows: "Seq Scan on Session (cost=0.00..2834.00 rows=50 ...)"
4
5-- Get actual row count
6SELECT COUNT(*) FROM "Session" WHERE "userId" = 'abc123';
7-- Returns: 350

Problem. Planner estimated 50 rows but actually there are 350 rows.

Why. Statistics are stale, the Session table has 2,770 modifications since last analyze (44 days ago). User session distribution has changed significantly since last ANALYZE.

Impact.

  • Planner might choose nested loop join expecting small result set
  • Allocated memory might be insufficient
  • Query takes significantly longer than it should
  • If this query is used in joins, the entire execution plan could be suboptimal

Solution.

1ANALYZE "Session";

After running ANALYZE, check the plan again.

1EXPLAIN SELECT * FROM "Session" WHERE "userId" = 'abc123';
2-- Now shows: "Seq Scan on Session (cost=0.00..2834.00 rows=345 ...)"

Much better! The estimate is now close to the actual count (345 vs 350).

Common Solutions in PostgreSQL

2.3.1.

Manual ANALYZE

1-- Analyze specific table

3
4-- Analyze specific column (faster for large tables)
5ANALYZE "Notification_Logging" ("userId", "createdAt");
6
7-- Analyze all tables in database
8ANALYZE;
9
10-- Verbose mode to see what's happening
11ANALYZE VERBOSE "Session";
2.3.2.

Configure Autovacuum

1-- Per-table autovacuum settings
2ALTER TABLE "Notification_Logging" SET (
3  autovacuum_analyze_scale_factor = 0.05,  -- Analyze after 5% of rows change
4  autovacuum_analyze_threshold = 1000      -- Minimum 1000 rows before analyzing
5);
6
7-- Check current autovacuum settings
8SELECT 
9  relname,
10  reloptions
11FROM pg_class
12WHERE relname = 'Notification_Logging';

Recommended settings for high-traffic tables:

  • autovacuum_analyze_scale_factor = 0.05 (5% threshold instead of default 10%)

  • autovacuum_analyze_threshold = 500 (lower threshold for more frequent updates)

3.

Stale Statistics in Oracle Database

3.1.

How Statistics Affect Execution Plans

Oracle's Cost-Based Optimizer (CBO) relies on statistics stored in the data dictionary. Stale statistics can lead to.

  • Wrong join order selection
  • Incorrect decision between index scan and full table scan
  • Poor cardinality estimates
  • Suboptimal partition pruning

3.2.

Detecting Stale Statistics

3.2.1.

Check DBA_TAB_STATISTICS

1SELECT 
2  owner,
3  table_name,
4  num_rows,
5  last_analyzed,
6  stale_stats
7FROM dba_tab_statistics
8WHERE owner NOT IN ('SYS', 'SYSTEM')
9  AND stale_stats = 'YES'
10ORDER BY last_analyzed;

Key indicators.

  • stale_stats = 'YES' - Oracle automatically marks statistics as stale when > 10% of rows change
  • last_analyzed shows when statistics were last gathered
  • num_rows = NULL means statistics have never been gathered
3.2.2.

Query execution plan analysis

1-- Get execution plan with cost estimates
2EXPLAIN PLAN FOR
3SELECT * FROM orders WHERE customer_id = 12345;
4
5-- View the plan
6SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

Look for:

  • Large discrepancy between estimated rows and actual rows
  • Full table scans where index scans should be used
  • Hash joins where nested loops would be better (or vice versa)

3.3.

Solutions in Oracle Database

3.3.1.

Oracle's Schema Ownership Model

In oracle we are also able to update the statistic, but to understand the parameters used in that SQL command, we need to understand the models in Oracle database.

In Oracle, each user is also a schema.

Oracle model.

1-- When we create a user, a schema with the same name is automatically created
2CREATE USER sales_user IDENTIFIED BY password;
3-- This creates:
4-- - User: sales_user (can login)
5-- - Schema: sales_user (namespace for objects)
6
7-- When sales_user creates a table, it's in their own schema
8CONNECT sales_user/password;
9CREATE TABLE orders (...);  -- Creates sales_user.orders
10
11-- The user owns ALL objects in their schema
12-- Schema name = User name (always)

PostgreSQL model (for comparison).

1-- In PostgreSQL, schemas are separate from users
2CREATE USER sales_user WITH PASSWORD 'password';
3CREATE USER inventory_user WITH PASSWORD 'password';
4
5-- Schemas are independent containers
6CREATE SCHEMA sales_schema;
7CREATE SCHEMA inventory_schema;
8
9-- Different users can own objects in the same schema
10CREATE TABLE sales_schema.orders (...);  -- Can be owned by sales_user
11ALTER TABLE sales_schema.orders OWNER TO sales_user;
12
13CREATE TABLE sales_schema.customers (...);  -- Can be owned by inventory_user  
14ALTER TABLE sales_schema.customers OWNER TO inventory_user;
15
16-- Schema ≠ User  (separate concepts)

Key differences.

AspectOraclePostgreSQL
Schema creationAutomatic (when user created)Explicit (CREATE SCHEMA)
Schema ownershipOne schema = one user owns ALL objects in itMultiple users can own different tables in the same schema
NamingSchema name = User nameSchema names independent of users
Schema vs UserSame entity (1-to-1)Different entities (many-to-many)
Object ownershipAll objects in schema owned by schema's userDifferent users can own different objects in same schema

Simply put:

  • In oracle all tables belong to the schema = user.

  • While in other database such as PostgreSQL, tables still belong to one schema, but different users can own different tables in that schema

    Therefore before we can remove a user, we need to transfer the owner of that table to other user first, usually the pgadmin user (again, in pgsql, role = user)).

Oracle Statistics Collection Context.

When we use DBMS_STATS.GATHER_SCHEMA_STATS:

1-- Gather stats for sales_user schema
2BEGIN
3  DBMS_STATS.GATHER_SCHEMA_STATS(
4    ownname => 'SALES_USER',  -- This is both the schema AND the owner
5    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
6    cascade => TRUE
7  );
8END;
9/

What ownname really means in Oracle:

  • ownname = Schema name = Owner name (all the same thing!)
  • It's not asking "who owns the tables" (the schema always owns its own tables)
  • It's asking "which schema (user's namespace) should we analyze?"

Example with multiple users/schemas:

1-- Create three users (three schemas automatically created)
2CREATE USER sales_user IDENTIFIED BY pass1;
3CREATE USER inventory_user IDENTIFIED BY pass2;
4CREATE USER hr_user IDENTIFIED BY pass3;
5
6-- Each user creates tables in their own schema
7CONNECT sales_user/pass1;
8CREATE TABLE orders (...);      -- Creates sales_user.orders
9
10CONNECT inventory_user/pass2;
11CREATE TABLE products (...);    -- Creates inventory_user.products
12
13CONNECT hr_user/pass3;
14CREATE TABLE employees (...);   -- Creates hr_user.employees
15
16-- To gather statistics for each schema:
17CONNECT system/system_password;
18
19-- Gather stats for sales_user's schema
20EXEC DBMS_STATS.GATHER_SCHEMA_STATS('SALES_USER');
21
22-- Gather stats for inventory_user's schema
23EXEC DBMS_STATS.GATHER_SCHEMA_STATS('INVENTORY_USER');
24
25-- Gather stats for hr_user's schema
26EXEC DBMS_STATS.GATHER_SCHEMA_STATS('HR_USER');

Question. Why the parameter is called ownname?

Historical naming - in Oracle's implementation, the "owner" of all objects in a schema is the user that the schema belongs to. Since schema = user in Oracle, the parameter name ownname essentially means "which user's schema".

PostgreSQL equivalent (for comparison).

1-- In PostgreSQL, schemas are separate from users
2-- Multiple users can own tables in the same schema
3
4-- Analyze specific tables in a schema
5ANALYZE sales_schema.orders;
6ANALYZE sales_schema.order_items;
7
8-- Analyze all tables in a specific schema
9ANALYZE sales_schema;
10
11-- Analyze all tables owned by a specific user across multiple schemas
12DO $$
13DECLARE
14  tbl record;
15BEGIN
16  FOR tbl IN 
17    SELECT schemaname, tablename 
18    FROM pg_tables 
19    WHERE tableowner = 'sales_user'
20  LOOP
21    EXECUTE format('ANALYZE %I.%I', tbl.schemaname, tbl.tablename);
22  END LOOP;
23END $$;

Summary.

  • In Oracle: ownname = schema name = user name (all same)
  • In PostgreSQL: user ≠ schema (different users can own tables in same schema)
  • Oracle's DBMS_STATS.GATHER_SCHEMA_STATS('SALES_USER') means "gather stats for all tables in the SALES_USER schema"
  • PostgreSQL's ANALYZE sales_schema updates stats for all tables in that schema, regardless of who owns them

3.3.3.

Understanding GATHER Command Options

Oracle provides different gathering strategies through the options parameter.

Available options.

1BEGIN
2  DBMS_STATS.GATHER_SCHEMA_STATS(
3    ownname => 'HR',
4    options => 'GATHER STALE',  -- Only gather stale statistics
5    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE
6  );
7END;
8/

Option values.

OptionDescriptionWhen to Use
GATHERGather stats even if not staleAfter major data changes
GATHER STALEOnly gather if marked staleRegular maintenance
GATHER EMPTYOnly gather if no stats existNew tables only
GATHER AUTOOracle decides based on stalenessRecommended default

Decision guide for choosing the right option:

3.3.4.

Understand the different GATHER commands

Three GATHER commands serve different purposes:

CommandDescription
GATHER_TABLE_STATSPurpose. Update statistics for one specific table

When to use. One table has stale statistics, after large data changes to a single table, quick targeted fix

Duration. Seconds to minutes (one table only)
GATHER_SCHEMA_STATSPurpose. Update statistics for all tables in a schema at once

When to use. Multiple tables are stale, after bulk data load affecting many tables, regular maintenance

Duration. Minutes to hours (depends on schema size)

Note. Automatically gathers stats for all tables, so we don't need individual GATHER_TABLE_STATS for each table
GATHER_SYSTEM_STATSPurpose. Collects hardware performance metrics (CPU speed, I/O throughput, disk latency)

When to use. Once after hardware changes or database migration, helps optimizer understand system capabilities

Duration. Hours (needs workload sampling)

Note. This is NOT about table statistics - it's about hardware performance characteristics

Examples for different scenarios.

1-- Scenario 1: Just loaded 1M rows into orders table
2BEGIN
3  DBMS_STATS.GATHER_TABLE_STATS(
4    ownname => 'SALES',
5    tabname => 'ORDERS',
6    options => 'GATHER',  -- Force gather even if not marked stale
7    cascade => TRUE
8  );
9END;
10/
11
12-- Scenario 2: Daily maintenance job
13BEGIN
14  DBMS_STATS.GATHER_SCHEMA_STATS(
15    ownname => 'SALES',
16    options => 'GATHER STALE',  -- Only tables that changed >10%
17    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE
18  );
19END;
20/
21
22-- Scenario 3: New database setup
23BEGIN
24  DBMS_STATS.GATHER_DATABASE_STATS(
25    options => 'GATHER EMPTY',  -- Only tables with no stats
26    estimate_percent => 100  -- Analyze all rows for accuracy
27  );
28END;
29/
30
31-- Scenario 4: Unsure what's needed
32BEGIN
33  DBMS_STATS.GATHER_DATABASE_STATS(
34    options => 'GATHER AUTO',  -- Let Oracle decide
35    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE
36  );
37END;

4.

Common Practices Across All Databases

4.1.

Regular Maintenance Schedule

  • PostgreSQL. Enable autovacuum and schedule periodic manual ANALYZE during maintenance windows
  • Oracle. Use automatic statistics collection and supplement with manual gathering after large data changes

4.2.

Monitor After Bulk Operations

All databases require statistics updates after.

  • Large INSERT/UPDATE/DELETE operations
  • Data imports or migrations
  • Index creation or modification
  • Schema changes

4.3.

Test Query Plans

Always verify execution plans in development before deploying.

1-- PostgreSQL
2EXPLAIN ANALYZE SELECT ...;
3
4-- Oracle
5EXPLAIN PLAN FOR SELECT ...;
6SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

4.4.

Set Up Alerts

Monitor for performance degradation that might indicate stale statistics:

  • Sudden increases in query execution time
  • Changes in execution plan choices
  • Resource utilization spikes
4.4.1.

PostgreSQL alert setup

1-- Using pg_stat_statements extension
2CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
3
4-- Query to find slow queries (run periodically via monitoring tool)
5SELECT 
6  query,
7  calls,
8  mean_exec_time,
9  stddev_exec_time,
10  (total_exec_time / 1000 / 60) as total_minutes
11FROM pg_stat_statements
12WHERE mean_exec_time > 1000  -- Alert if mean time > 1 second
13ORDER BY mean_exec_time DESC
14LIMIT 20;
15
16-- Set up alert via cron + monitoring system (e.g., Prometheus, Grafana)
17-- Or use PostgreSQL log_min_duration_statement
18ALTER SYSTEM SET log_min_duration_statement = 1000;  -- Log queries > 1s
19SELECT pg_reload_conf();
4.4.2.

Oracle alert setup

1-- Using Oracle Enterprise Manager or custom monitoring
2-- Create alert for SQL execution time variance
3
4-- 1. Baseline query performance
5BEGIN
6  DBMS_WORKLOAD_REPOSITORY.CREATE_BASELINE(
7    start_snap_id => 1000,
8    end_snap_id   => 1100,
9    baseline_name => 'NORMAL_WORKLOAD',
10    expiration    => NULL
11  );
12END;
13/
14
15-- 2. Monitor for queries exceeding baseline by threshold
16SELECT 
17  sql_id,
18  executions,
19  elapsed_time/1000000 as elapsed_seconds,
20  cpu_time/1000000 as cpu_seconds,
21  buffer_gets,
22  disk_reads
23FROM v$sql
24WHERE elapsed_time/executions > 1000000  -- > 1 second average
25  AND executions > 10
26ORDER BY elapsed_time DESC;
27
28-- 3. Set up alert via DBMS_SERVER_ALERT
29BEGIN
30  DBMS_SERVER_ALERT.SET_THRESHOLD(
31    metrics_id              => DBMS_SERVER_ALERT.SQL_ELAPSED_TIME,
32    warning_operator        => DBMS_SERVER_ALERT.OPERATOR_GT,
33    warning_value           => '5000',  -- 5 seconds
34    critical_operator       => DBMS_SERVER_ALERT.OPERATOR_GT,
35    critical_value          => '10000', -- 10 seconds
36    observation_period      => 1,
37    consecutive_occurrences => 1,
38    instance_name           => NULL,
39    object_type             => NULL,
40    object_name             => NULL
41  );
42END;
43/
